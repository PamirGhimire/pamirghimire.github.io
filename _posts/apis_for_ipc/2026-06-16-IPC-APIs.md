In embedded software and elsewhere, when two processes need to talk to each other, they need primarily two things:
1. An Interface specification: This mainly describes the datatypes that will be exchanged, and methods that will be called on the interface provider by the interface user. In embedded world, an interface is usually provided by one process (which may be called the server), and consumed by others (clients) which require it.  
 2. A transport mechanism: Here, we need a structured way of talking between the processes, and a physical possibility to do so. The structured way may consist of several protocols along the OSI layers - like SOME/IP, GRPC, HTTP, etc. on application level, UDP or TCP on transport level, Ethernet on network level, and WiFi/Bluetooth/Ethernet on physical level. If the processes are on the same physical machine, they could also use shared memory as the physical medium for exchanging messages together with one of the protocols, like in case of socket communication over localhost. 

 To try and understand these two aspects of IPC, and lead into some finer aspects, let us start with a very simple example consisting of one provider which emits an integer value, and one consumer that consumes it. The producer could be for example the wheel ticks rotary encoder on a vehicle, and the consumer a software odometer that consumes this value and calculates how far the wheel has rotated.
